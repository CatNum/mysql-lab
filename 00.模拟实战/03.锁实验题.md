# 03.锁实验题

> 目标：通过两个或多个 MySQL 会话观察 InnoDB 行锁、锁等待、间隙锁、临键锁、唯一索引冲突等待、死锁，以及普通 SELECT 和当前读的差异。完整原理答案放在 `06.标准答案与观测指标.md`。

## 0. 通用观察命令

```sql
SHOW PROCESSLIST;
SHOW ENGINE INNODB STATUS\G

SELECT *
FROM performance_schema.data_locks\G

SELECT *
FROM performance_schema.data_lock_waits\G
```

字段说明：

- `stock`：库存数量，用于模拟扣减库存时的并发更新。
- `status`：订单状态，用于模拟订单状态流转。
- `id`：主键 ID，用于精确命中单行并观察行锁。
- `created_at`：创建时间，用于范围更新、间隙锁和临键锁实验。

## LOCK-01：库存扣减行锁

### 场景

两个支付请求同时扣减热点商品 `products.id = 2001` 的库存。

### 目标

观察主键等值更新命中单行时的行锁和锁等待。

### 会话 A

```sql
USE interview_lab;
SET autocommit = 0;
START TRANSACTION;

UPDATE products
SET stock = stock - 1
WHERE id = 2001 AND stock > 0;

-- 暂时不要 COMMIT，保持事务打开。
```

### 会话 B

```sql
USE interview_lab;
SET autocommit = 0;
SET innodb_lock_wait_timeout = 10;
START TRANSACTION;

UPDATE products
SET stock = stock - 1
WHERE id = 2001 AND stock > 0;

-- 观察是否阻塞。等会话 A COMMIT 后再看结果。
```

### 观察命令

```sql
SHOW PROCESSLIST;
SHOW ENGINE INNODB STATUS\G
SELECT * FROM performance_schema.data_lock_waits\G
```

### 预期现象

会话 B 被阻塞，直到会话 A `COMMIT` 或 `ROLLBACK`。

### 思考问题

1. 为什么 `WHERE id = 2001` 会锁定单行？
2. `stock > 0` 是业务条件还是锁范围条件？
3. 如果没有 `stock > 0` 会有什么风险？
4. 高并发扣库存除了数据库行锁，还有哪些方案？

## LOCK-02：订单状态更新锁等待

### 场景

支付回调和后台取消订单同时修改 `orders.id = 3001`。

### 目标

观察同一订单状态更新时的锁等待，以及状态机条件的重要性。

### 会话 A

```sql
USE interview_lab;
SET autocommit = 0;
START TRANSACTION;

UPDATE orders
SET status = 1, pay_type = 1, pay_time = NOW(), updated_at = NOW()
WHERE id = 3001 AND status = 0;

-- 不提交，模拟支付回调处理中。
```

### 会话 B

```sql
USE interview_lab;
SET autocommit = 0;
SET innodb_lock_wait_timeout = 10;
START TRANSACTION;

UPDATE orders
SET status = 4, updated_at = NOW()
WHERE id = 3001 AND status = 0;
```

### 观察命令

```sql
SHOW PROCESSLIST;
SHOW ENGINE INNODB STATUS\G
SELECT * FROM performance_schema.data_locks\G
SELECT * FROM performance_schema.data_lock_waits\G
```

### 预期现象

会话 B 等待会话 A 的行锁。会话 A 提交后，会话 B 继续执行，但可能因为 `status = 0` 条件不再满足而影响 0 行。

### 思考问题

1. 为什么更新订单状态要带旧状态条件？
2. 锁等待结束后，MySQL 会不会重新判断 `WHERE` 条件？
3. 如何保证订单状态流转幂等？
4. 业务上如何处理 affected rows = 0？

## LOCK-03：范围更新触发间隙锁或临键锁

### 场景

后台批处理按创建时间范围扫描并更新订单。

### 目标

在 REPEATABLE READ 下观察范围更新可能产生 next-key lock，导致范围内插入被阻塞。

### 会话 A

```sql
USE interview_lab;
SET autocommit = 0;
SET SESSION TRANSACTION ISOLATION LEVEL REPEATABLE READ;
START TRANSACTION;

UPDATE orders FORCE INDEX (idx_created_at)
SET updated_at = NOW()
WHERE created_at >= '2026-01-01 00:00:00'
  AND created_at <  '2026-01-02 00:00:00';

-- 不提交，保持范围锁。
```

### 会话 B

```sql
USE interview_lab;
SET autocommit = 0;
SET SESSION TRANSACTION ISOLATION LEVEL REPEATABLE READ;
SET innodb_lock_wait_timeout = 10;
START TRANSACTION;

INSERT INTO orders(
  tenant_id, order_no, user_id, total_amount, status,
  pay_type, source, deleted, pay_time, created_at, updated_at
) VALUES (
  1, CONCAT('GAP_TEST_', UUID()), 1001, 88.88, 0,
  0, 1, 0, NULL, '2026-01-01 12:00:00', NOW()
);
```

### 观察命令

```sql
SHOW PROCESSLIST;
SHOW ENGINE INNODB STATUS\G
SELECT * FROM performance_schema.data_locks\G
SELECT * FROM performance_schema.data_lock_waits\G
```

### 预期现象

如果范围内存在可锁记录或索引范围被锁住，会话 B 的插入可能被阻塞，直到会话 A 提交或回滚。

### 思考问题

1. 什么是间隙锁？什么是临键锁？
2. 为什么范围条件比主键等值更新更容易扩大锁范围？
3. READ COMMITTED 下现象是否可能不同？
4. 如何降低批量范围更新对线上写入的影响？

## LOCK-04：唯一索引冲突等待

### 场景

两个注册请求同时使用同一个手机号。

### 目标

观察唯一索引冲突在事务未提交时的等待行为。

### 会话 A

```sql
USE interview_lab;
SET autocommit = 0;
START TRANSACTION;

INSERT INTO users(
  tenant_id, username, phone, email, status, deleted, city, age, created_at, updated_at
) VALUES (
  1, 'unique_wait_a', '13999990000', 'unique_wait_a@example.com', 1, 0, '杭州', 28, NOW(), NOW()
);

-- 不提交。
```

### 会话 B

```sql
USE interview_lab;
SET autocommit = 0;
SET innodb_lock_wait_timeout = 10;
START TRANSACTION;

INSERT INTO users(
  tenant_id, username, phone, email, status, deleted, city, age, created_at, updated_at
) VALUES (
  1, 'unique_wait_b', '13999990000', 'unique_wait_b@example.com', 1, 0, '杭州', 29, NOW(), NOW()
);
```

### 观察命令

```sql
SHOW PROCESSLIST;
SHOW ENGINE INNODB STATUS\G
SELECT * FROM performance_schema.data_lock_waits\G
```

### 预期现象

会话 B 等待会话 A。若会话 A 提交，会话 B 报唯一键冲突；若会话 A 回滚，会话 B 可继续插入成功。

### 思考问题

1. 为什么唯一键冲突不是立刻报错？
2. 未提交数据对唯一性检查有什么影响？
3. 注册接口应该如何处理重复手机号？
4. `INSERT IGNORE` 或 `ON DUPLICATE KEY UPDATE` 是否适合所有场景？

## LOCK-05：死锁复现

### 场景

两个事务按相反顺序更新两个商品库存。

### 目标

复现死锁，观察 InnoDB 如何检测并回滚其中一个事务。

### 会话 A

```sql
USE interview_lab;
SET autocommit = 0;
START TRANSACTION;

UPDATE products SET stock = stock - 1 WHERE id = 2001;

-- 等会话 B 更新 2002 后，再执行：
UPDATE products SET stock = stock - 1 WHERE id = 2002;
```

### 会话 B

```sql
USE interview_lab;
SET autocommit = 0;
START TRANSACTION;

UPDATE products SET stock = stock - 1 WHERE id = 2002;

-- 等会话 A 已经持有 2001 后，再执行：
UPDATE products SET stock = stock - 1 WHERE id = 2001;
```

### 观察命令

```sql
SHOW ENGINE INNODB STATUS\G
```

### 预期现象

其中一个会话报错：`Deadlock found when trying to get lock; try restarting transaction`。

### 思考问题

1. 死锁和普通锁等待有什么区别？
2. 为什么相反加锁顺序容易死锁？
3. 如何从 `SHOW ENGINE INNODB STATUS` 中读出死锁链路？
4. 业务代码如何降低死锁概率？

## LOCK-06：普通 SELECT、SELECT FOR UPDATE、UPDATE 的锁差异

### 场景

查询商品库存时，有的代码只是展示库存，有的代码要先锁库存再扣减。

### 目标

比较普通快照读、当前读和更新语句的锁差异。

### 会话 A

```sql
USE interview_lab;
SET autocommit = 0;
START TRANSACTION;

UPDATE products
SET stock = stock - 1
WHERE id = 2001;

-- 不提交。
```

### 会话 B

```sql
USE interview_lab;
SET autocommit = 0;
START TRANSACTION;

-- 普通 SELECT：通常不阻塞，读快照版本。
SELECT id, stock FROM products WHERE id = 2001;

-- 当前读：需要加锁，可能阻塞。
SELECT id, stock FROM products WHERE id = 2001 FOR UPDATE;
```

### 观察命令

```sql
SHOW PROCESSLIST;
SHOW ENGINE INNODB STATUS\G
SELECT * FROM performance_schema.data_lock_waits\G
```

### 预期现象

普通 `SELECT` 通常不阻塞；`SELECT ... FOR UPDATE` 会等待会话 A 的写锁。

### 思考问题

1. 普通 SELECT 为什么通常不加锁？
2. `SELECT ... FOR UPDATE` 属于快照读还是当前读？
3. UPDATE 读取的是旧版本还是最新可更新版本？
4. 什么场景必须使用当前读？
