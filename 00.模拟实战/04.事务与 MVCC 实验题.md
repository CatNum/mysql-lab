# 04.事务与 MVCC 实验题

> 目标：通过两个 MySQL 会话观察 READ COMMITTED、REPEATABLE READ、快照读、当前读、UPDATE 当前读、幻读和订单状态流转事务。完整答案放在 `06.标准答案与观测指标.md`。

## 0. 关键概念

- 快照读：普通 `SELECT` 基于 Read View 读取可见版本，不主动加锁。
- 当前读：`SELECT ... FOR UPDATE`、`UPDATE`、`DELETE` 读取最新可更新版本，并可能加锁。
- Read View：一致性读视图，用来判断哪个事务版本对当前事务可见。
- `READ COMMITTED`：读已提交，每条语句通常生成新的 Read View。
- `REPEATABLE READ`：可重复读，事务内普通快照读通常复用同一个 Read View。
- `status`：订单状态，用于观察事务更新前后的业务状态。

## TX-01：READ COMMITTED 下两次快照读看到不同结果

### 场景

客服后台查看订单 `orders.id = 3002` 状态，同时支付系统更新订单状态。

### 目标

观察 READ COMMITTED 下同一事务内两次普通 SELECT 可能看到不同已提交结果。

### 初始数据确认

```sql
USE interview_lab;
SELECT id, status FROM orders WHERE id = 3002;
```

### 会话 A

```sql
USE interview_lab;
SET SESSION TRANSACTION ISOLATION LEVEL READ COMMITTED;
SET autocommit = 0;
START TRANSACTION;

SELECT id, status FROM orders WHERE id = 3002;

-- 等会话 B 提交后再次查询。
SELECT id, status FROM orders WHERE id = 3002;

COMMIT;
```

### 会话 B

```sql
USE interview_lab;
SET autocommit = 0;
START TRANSACTION;

UPDATE orders
SET status = 3, updated_at = NOW()
WHERE id = 3002;

COMMIT;
```

### 预期现象

会话 A 第二次普通 SELECT 能看到会话 B 已提交的新状态。

### 思考问题

1. READ COMMITTED 为什么每次语句都可能读到新版本？
2. 这是否属于不可重复读？
3. 对账类业务是否适合 READ COMMITTED？
4. 如何恢复实验数据？

## TX-02：REPEATABLE READ 下两次快照读看到相同结果

### 场景

同一个事务内重复查看订单 `orders.id = 3002`。

### 目标

观察 REPEATABLE READ 下同一事务内普通 SELECT 的可重复读效果。

### 初始数据确认

```sql
UPDATE orders SET status = 1 WHERE id = 3002;
SELECT id, status FROM orders WHERE id = 3002;
```

### 会话 A

```sql
USE interview_lab;
SET SESSION TRANSACTION ISOLATION LEVEL REPEATABLE READ;
SET autocommit = 0;
START TRANSACTION;

SELECT id, status FROM orders WHERE id = 3002;

-- 等会话 B 提交后再次查询。
SELECT id, status FROM orders WHERE id = 3002;

COMMIT;
```

### 会话 B

```sql
USE interview_lab;
SET autocommit = 0;
START TRANSACTION;

UPDATE orders
SET status = 3, updated_at = NOW()
WHERE id = 3002;

COMMIT;
```

### 预期现象

会话 A 两次普通 SELECT 看到相同结果；提交后重新开启事务才看到新状态。

### 思考问题

1. REPEATABLE READ 的 Read View 何时生成？
2. 为什么事务内第二次普通 SELECT 看不到已提交更新？
3. 这和锁有没有直接关系？
4. 如果会话 A 使用 `SELECT ... FOR UPDATE` 会怎样？

## TX-03：快照读和当前读差异

### 场景

订单页面普通查询看到旧状态，但处理流程加锁读取时需要最新状态。

### 目标

比较普通 SELECT 和 `SELECT ... FOR UPDATE` 在同一事务中的可见性差异。

### 初始数据确认

```sql
UPDATE orders SET status = 1 WHERE id = 3002;
SELECT id, status FROM orders WHERE id = 3002;
```

### 会话 A

```sql
USE interview_lab;
SET SESSION TRANSACTION ISOLATION LEVEL REPEATABLE READ;
SET autocommit = 0;
START TRANSACTION;

-- 快照读，建立 Read View。
SELECT id, status FROM orders WHERE id = 3002;

-- 等会话 B 提交后，再执行普通 SELECT。
SELECT id, status FROM orders WHERE id = 3002;

-- 当前读，读取最新可更新版本并加锁。
SELECT id, status FROM orders WHERE id = 3002 FOR UPDATE;

COMMIT;
```

### 会话 B

```sql
USE interview_lab;
SET autocommit = 0;
START TRANSACTION;

UPDATE orders SET status = 3, updated_at = NOW()
WHERE id = 3002;

COMMIT;
```

### 预期现象

会话 A 的普通 SELECT 仍看到旧状态；`SELECT ... FOR UPDATE` 看到最新已提交状态。

### 思考问题

1. 为什么当前读不使用旧快照？
2. 当前读为什么要加锁？
3. 业务中哪些场景需要当前读？
4. 混用快照读和当前读时容易产生什么误解？

## TX-04：UPDATE 当前读

### 场景

订单支付状态被其他事务先更新，本事务之前的快照仍看到旧状态。

### 目标

验证 `UPDATE` 会基于最新可更新版本判断条件，而不是基于事务早期快照。

### 初始数据确认

```sql
UPDATE orders SET status = 0, pay_type = 0, pay_time = NULL WHERE id = 3001;
SELECT id, status FROM orders WHERE id = 3001;
```

### 会话 A

```sql
USE interview_lab;
SET SESSION TRANSACTION ISOLATION LEVEL REPEATABLE READ;
SET autocommit = 0;
START TRANSACTION;

-- 快照读看到待支付。
SELECT id, status FROM orders WHERE id = 3001;

-- 等会话 B 把状态改为已支付后执行：
UPDATE orders
SET status = 2, updated_at = NOW()
WHERE id = 3001 AND status = 1;

SELECT ROW_COUNT() AS affected_rows;
COMMIT;
```

### 会话 B

```sql
USE interview_lab;
SET autocommit = 0;
START TRANSACTION;

UPDATE orders
SET status = 1, pay_type = 1, pay_time = NOW(), updated_at = NOW()
WHERE id = 3001 AND status = 0;

COMMIT;
```

### 预期现象

会话 A 的 `UPDATE ... WHERE status = 1` 可以命中会话 B 已提交的新版本，说明 UPDATE 是当前读。

### 思考问题

1. UPDATE 为什么必须读取最新版本？
2. `ROW_COUNT()` 可以帮助判断什么？
3. 如果会话 A 的 WHERE 写成 `status = 0` 会怎样？
4. 状态机更新为什么要带旧状态条件？

## TX-05：幻读观察

### 场景

运营在事务中统计某个时间范围内的订单数量，另一个事务插入了符合条件的新订单。

### 目标

对比 READ COMMITTED 和 REPEATABLE READ 下普通快照读对“新增行”的可见性。

### 初始数据确认

```sql
SELECT COUNT(*)
FROM orders
WHERE user_id = 1001
  AND created_at >= '2026-01-01 00:00:00'
  AND created_at <  '2026-01-02 00:00:00';
```

### 会话 A：READ COMMITTED 版本

```sql
USE interview_lab;
SET SESSION TRANSACTION ISOLATION LEVEL READ COMMITTED;
SET autocommit = 0;
START TRANSACTION;

SELECT COUNT(*) AS c1
FROM orders
WHERE user_id = 1001
  AND created_at >= '2026-01-01 00:00:00'
  AND created_at <  '2026-01-02 00:00:00';

-- 等会话 B 插入并提交后再次查询。
SELECT COUNT(*) AS c2
FROM orders
WHERE user_id = 1001
  AND created_at >= '2026-01-01 00:00:00'
  AND created_at <  '2026-01-02 00:00:00';

COMMIT;
```

### 会话 A：REPEATABLE READ 版本

```sql
USE interview_lab;
SET SESSION TRANSACTION ISOLATION LEVEL REPEATABLE READ;
SET autocommit = 0;
START TRANSACTION;

SELECT COUNT(*) AS c1
FROM orders
WHERE user_id = 1001
  AND created_at >= '2026-01-01 00:00:00'
  AND created_at <  '2026-01-02 00:00:00';

-- 等会话 B 插入并提交后再次查询。
SELECT COUNT(*) AS c2
FROM orders
WHERE user_id = 1001
  AND created_at >= '2026-01-01 00:00:00'
  AND created_at <  '2026-01-02 00:00:00';

COMMIT;
```

### 会话 B

```sql
USE interview_lab;
INSERT INTO orders(
  tenant_id, order_no, user_id, total_amount, status,
  pay_type, source, deleted, pay_time, created_at, updated_at
) VALUES (
  1, CONCAT('PHANTOM_', UUID()), 1001, 66.66, 0,
  0, 1, 0, NULL, '2026-01-01 12:34:56', NOW()
);
```

### 预期现象

READ COMMITTED 下第二次统计可能变大；REPEATABLE READ 下普通快照读通常保持不变。

### 思考问题

1. 幻读和不可重复读有什么区别？
2. REPEATABLE READ 普通快照读为什么能避免看到幻影行？
3. `SELECT ... FOR UPDATE` 的范围锁行为是否不同？
4. 为什么不能简单说“RR 一定完全没有幻读问题”？

## TX-06：订单状态流转事务

### 场景

支付成功后，需要把订单从待支付改为已支付，并写入订单日志。

### 目标

练习真实业务中的事务边界、状态机条件、幂等和日志一致性。

### 初始数据确认

```sql
UPDATE orders SET status = 0, pay_type = 0, pay_time = NULL WHERE id = 3001;
DELETE FROM order_logs WHERE order_id = 3001 AND remark LIKE 'TX-06%';

SELECT id, status, pay_time FROM orders WHERE id = 3001;
```

### 会话 A

```sql
USE interview_lab;
SET autocommit = 0;
START TRANSACTION;

UPDATE orders
SET status = 1,
    pay_type = 1,
    pay_time = NOW(),
    updated_at = NOW()
WHERE id = 3001
  AND status = 0;

SELECT ROW_COUNT() AS affected_rows;

INSERT INTO order_logs(tenant_id, order_id, old_status, new_status, remark, created_at)
SELECT tenant_id, id, 0, 1, 'TX-06 支付成功', NOW()
FROM orders
WHERE id = 3001;

COMMIT;
```

### 会话 B

```sql
USE interview_lab;
SET autocommit = 0;
START TRANSACTION;

UPDATE orders
SET status = 4,
    updated_at = NOW()
WHERE id = 3001
  AND status = 0;

SELECT ROW_COUNT() AS affected_rows;

COMMIT;
```

### 预期现象

只有一个事务能成功把订单从 `status = 0` 改走；另一个事务要么等待后影响 0 行，要么根据业务逻辑放弃处理。

### 思考问题

1. 订单状态流转为什么必须在事务内完成？
2. 先写日志还是先更新订单？为什么？
3. 如何处理重复支付回调？
4. 如何用 `ROW_COUNT()` 做幂等判断？
