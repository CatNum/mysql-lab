# 05.慢查询排查实验题

> 目标：模拟线上慢查询排查。每道题都按照“现象 -> 问题 SQL -> 第一轮定位 -> 观察指标 -> 临时止血 -> 长期优化 -> 面试追问”的结构展开。完整标准答案放在 `06.标准答案与观测指标.md`。

## 0. 慢查询排查通用路径

1. 先定位：确认影响面、慢 SQL、参数分布、执行计划。
2. SQL 写法优化：减少 `SELECT *`、避免函数包裹索引列、避免隐式类型转换。
3. 索引优化：补联合索引、覆盖索引、调整索引顺序。
4. 查询路径优化：游标分页、延迟回表、预聚合、拆查询。
5. 缓存或异步化：缓存热点结果、异步统计报表。
6. 架构优化：读写分离、分区、分表、分库分表。

观察指标：

- `type`：访问类型。
- `key`：实际使用索引。
- `rows`：预估扫描行数。
- `filtered`：过滤比例。
- `Extra`：是否出现 `Using filesort`、`Using temporary`、`Using index`、`Using index condition`。
- 实际耗时：可以通过 `EXPLAIN ANALYZE`、慢日志、应用日志观察。
- 扫描行数 / 返回行数：扫描很多但返回很少通常说明查询路径不理想。

## SLOW-01：订单列表深分页突然变慢

### 线上现象

用户反馈“我的订单”翻到后面页数明显变慢。接口从几十毫秒变成数秒，热点用户更明显。

### 问题 SQL

```sql
SELECT *
FROM orders
WHERE user_id = 1001
ORDER BY created_at DESC
LIMIT 10000, 20;
```

### 第一轮定位

```sql
EXPLAIN SELECT *
FROM orders
WHERE user_id = 1001
ORDER BY created_at DESC
LIMIT 10000, 20;

EXPLAIN ANALYZE SELECT *
FROM orders
WHERE user_id = 1001
ORDER BY created_at DESC
LIMIT 10000, 20;
```

### 观察指标

记录 `type`、`key`、`rows`、`Extra`、实际耗时、扫描行数、返回行数。

### 临时止血

- 限制最大可翻页页数。
- 将 `SELECT *` 改为只返回列表页必需字段。
- 对热点用户订单列表做短期缓存。

### 长期优化

1. 改为游标分页：`WHERE user_id = ? AND (created_at, id) < (?, ?) ORDER BY created_at DESC, id DESC LIMIT 20`。
2. 创建更合适的索引：`(user_id, created_at, id)` 或结合状态创建 `(user_id, status, created_at, id)`。
3. 使用延迟回表：先查主键 ID，再回表取详情。
4. 对历史订单归档或分区，降低活跃表扫描压力。

### 面试追问

1. 为什么 `LIMIT 10000,20` 不是只读 20 行？
2. 游标分页的缺点是什么？
3. 为什么列表页不建议 `SELECT *`？
4. 延迟回表适合什么场景？

## SLOW-02：热点租户订单列表扫描行数过大

### 线上现象

大客户租户 `tenant_id = 1` 的订单列表变慢，普通租户正常。

### 问题 SQL

```sql
SELECT id, order_no, tenant_id, user_id, status, total_amount, created_at
FROM orders
WHERE tenant_id = 1
  AND status = 3
ORDER BY created_at DESC
LIMIT 50;
```

### 第一轮定位

```sql
EXPLAIN SELECT id, order_no, tenant_id, user_id, status, total_amount, created_at
FROM orders
WHERE tenant_id = 1
  AND status = 3
ORDER BY created_at DESC
LIMIT 50;

SELECT tenant_id, COUNT(*) AS c
FROM orders
GROUP BY tenant_id
ORDER BY c DESC
LIMIT 10;
```

### 观察指标

关注热点租户数据量、`key` 是否使用 `idx_tenant_created_at`、`rows` 是否过大、`status` 是否在索引中被充分利用。

### 临时止血

- 增加时间范围条件，例如只查询最近 90 天。
- 限制页面最大翻页深度。
- 对热点租户订单列表做缓存或读库隔离。

### 长期优化

1. 补充联合索引：`ALTER TABLE orders ADD INDEX idx_tenant_status_created_id (tenant_id, status, created_at, id);`
2. 控制返回字段，避免大字段和无关字段回表。
3. 热点租户单独做数据分区、归档或分表。
4. 高频状态统计可异步写入汇总表。

### 面试追问

1. 为什么同一条 SQL 在不同租户下耗时差异很大？
2. 数据倾斜如何影响优化器估算？
3. `tenant_id`、`status`、`created_at` 的索引顺序如何设计？
4. 热点租户是否一定要分库分表？

## SLOW-03：订单日志按时间范围查询变慢

### 线上现象

运维查询最近一段时间的订单日志，随着日志表增长，查询越来越慢。

### 问题 SQL

```sql
SELECT *
FROM order_logs
WHERE created_at >= DATE_SUB(NOW(), INTERVAL 30 DAY)
ORDER BY created_at DESC
LIMIT 10000, 100;
```

### 第一轮定位

```sql
EXPLAIN SELECT *
FROM order_logs
WHERE created_at >= DATE_SUB(NOW(), INTERVAL 30 DAY)
ORDER BY created_at DESC
LIMIT 10000, 100;

EXPLAIN ANALYZE SELECT *
FROM order_logs
WHERE created_at >= DATE_SUB(NOW(), INTERVAL 30 DAY)
ORDER BY created_at DESC
LIMIT 10000, 100;
```

### 观察指标

关注 `idx_created_at` 是否使用、`rows` 是否接近 offset、`SELECT *` 是否导致大量回表。

### 临时止血

- 限制查询时间范围。
- 禁止深分页导出，改成异步任务。
- 只返回日志 ID、订单 ID、状态和创建时间等必要字段。

### 长期优化

1. 改游标分页：基于 `created_at` 和 `id` 继续向后拉取。
2. 创建覆盖索引：例如 `(created_at, id, order_id, new_status)` 支撑列表页。
3. 日志表按时间归档、分区或冷热分离。
4. 运维导出走离线库或数仓，不直接打线上主库。

### 面试追问

1. 日志表为什么特别容易出现深分页问题？
2. `created_at` 单列索引为什么不一定够？
3. 归档和分区分别解决什么问题？
4. 为什么导出类查询要异步化？

## SLOW-04：商品销量报表 JOIN 和 GROUP BY 变慢

### 线上现象

运营后台查看近 30 天商品销量排行榜，接口偶发超时。

### 问题 SQL

```sql
SELECT oi.product_id,
       SUM(oi.quantity) AS total_quantity,
       SUM(oi.quantity * oi.price) AS total_amount
FROM order_items oi
JOIN orders o ON o.id = oi.order_id
WHERE o.created_at >= DATE_SUB(NOW(), INTERVAL 30 DAY)
  AND o.status = 3
GROUP BY oi.product_id
ORDER BY total_quantity DESC
LIMIT 20;
```

### 第一轮定位

```sql
EXPLAIN SELECT oi.product_id,
       SUM(oi.quantity) AS total_quantity,
       SUM(oi.quantity * oi.price) AS total_amount
FROM order_items oi
JOIN orders o ON o.id = oi.order_id
WHERE o.created_at >= DATE_SUB(NOW(), INTERVAL 30 DAY)
  AND o.status = 3
GROUP BY oi.product_id
ORDER BY total_quantity DESC
LIMIT 20;
```

### 观察指标

关注 JOIN 驱动表、`rows`、是否出现 `Using temporary`、`Using filesort`，以及 `order_items` 是否大量扫描。

### 临时止血

- 缩小时间范围。
- 降低查询频率，加后台缓存。
- 将报表从实时查询改为手动刷新。

### 长期优化

1. 为订单过滤补索引：`orders(status, created_at, id)`。
2. 为明细关联和聚合补索引：根据查询路径考虑 `order_items(order_id, product_id)` 或 `order_items(product_id, created_at)`。
3. 建商品日销量汇总表，订单完成后异步累计。
4. 报表迁移到 OLAP、数仓或搜索分析系统。

### 面试追问

1. 为什么 JOIN + GROUP BY + ORDER BY 容易慢？
2. 为什么不能只说“给 product_id 加索引”？
3. 实时报表和离线报表边界怎么划分？
4. 临时表和 filesort 说明了什么？

## SLOW-05：隐式类型转换导致手机号查询异常

### 线上现象

登录接口偶尔慢，排查发现手机号字段是字符串，但代码传参有时是数字。

### 问题 SQL

```sql
SELECT id, username, phone
FROM users
WHERE phone = 13800001001;
```

### 第一轮定位

```sql
EXPLAIN SELECT id, username, phone
FROM users
WHERE phone = 13800001001;

SHOW WARNINGS;

EXPLAIN SELECT id, username, phone
FROM users
WHERE phone = '13800001001';
```

### 观察指标

对比两条 SQL 的 `type`、`key`、`rows`，观察是否稳定使用 `uk_phone`。

### 临时止血

- 立即修复接口参数类型，手机号统一作为字符串。
- 增加参数校验和日志报警。

### 长期优化

1. 统一接口契约和 ORM 字段类型。
2. 增加 SQL 审核规则，禁止字段类型不匹配比较。
3. 对核心登录 SQL 加入慢查询巡检。
4. 保持手机号字段为字符串，避免前导 0、国际区号和超出整数范围问题。

### 面试追问

1. 隐式类型转换为什么可能破坏索引？
2. 字段侧转换和常量侧转换有什么区别？
3. 为什么手机号不建议用 BIGINT？
4. 如何从慢日志中识别这类问题？

## SLOW-06：锁等待导致 SQL 看起来变慢

### 线上现象

一条按主键更新商品库存的 SQL 偶尔执行很久，但执行计划显示主键命中，扫描行数很小。

### 问题 SQL

```sql
UPDATE products
SET stock = stock - 1
WHERE id = 2001 AND stock > 0;
```

### 第一轮定位

```sql
SHOW PROCESSLIST;
SHOW ENGINE INNODB STATUS\G
SELECT * FROM performance_schema.data_locks\G
SELECT * FROM performance_schema.data_lock_waits\G
```

### 观察指标

关注 SQL 状态是否为等待锁、阻塞源事务是谁、阻塞持续多久、是否存在长事务。

### 临时止血

- 找到阻塞事务，确认后提交、回滚或 kill。
- 降低扣库存接口并发，临时排队。
- 缩短事务持有时间，避免事务内调用外部服务。

### 长期优化

1. 规范事务边界，先算好数据再开启事务。
2. 固定加锁顺序，降低死锁概率。
3. 热点库存使用队列、Redis 预扣、分段库存或异步削峰。
4. 建立长事务和锁等待监控报警。

### 面试追问

1. 执行计划很好为什么 SQL 仍然慢？
2. 如何区分 CPU/IO 慢和锁等待慢？
3. 长事务对 InnoDB 有哪些影响？
4. 热点行更新有哪些治理方案？
