# 06.标准答案与观测指标

> 目标：汇总 `02.索引实验题.md`、`03.锁实验题.md`、`04.事务与 MVCC 实验题.md`、`05.慢查询排查实验题.md` 的标准答案、观测指标和面试表达。本文适合复习和面试背诵，不写成数据库教材。

## 1. 观测指标总表

| 类别 | 指标 / 命令 | 含义 | 面试判断标准 |
|---|---|---|---|
| EXPLAIN | `type` | 访问类型，表示 MySQL 访问表的方式，如 `ref`、`range`、`index`、`ALL` | `ALL` 通常要警惕；`ref/range` 需要结合 rows 判断 |
| EXPLAIN | `key` | 优化器实际选择的索引 | 是否命中符合查询条件和排序条件的索引 |
| EXPLAIN | `rows` | 预估扫描行数 | 扫描远大于返回行数，通常说明索引或查询路径不理想 |
| EXPLAIN | `filtered` | 存储引擎取出数据后预计保留比例 | 过滤比例低说明大量数据被取出后又丢弃 |
| EXPLAIN | `Extra` | 额外执行信息 | 重点看 `Using index`、`Using index condition`、`Using filesort`、`Using temporary` |
| 执行分析 | `EXPLAIN ANALYZE` | 真实执行分析，展示实际耗时和实际行数 | 用于验证优化器估算和真实执行差异 |
| 线程状态 | `SHOW PROCESSLIST` | 查看连接、线程状态和正在执行的 SQL | 发现锁等待、长事务、慢 SQL、Sleep 连接 |
| InnoDB 状态 | `SHOW ENGINE INNODB STATUS` | 查看 InnoDB 事务、锁等待、死锁信息 | 死锁排查核心命令 |
| 锁信息 | `performance_schema.data_locks` | 当前锁信息表 | 查看锁类型、锁对象、持锁事务 |
| 锁等待 | `performance_schema.data_lock_waits` | 当前锁等待关系 | 建立谁阻塞谁的链路 |
| 慢查询 | 慢日志 | 记录超过阈值的 SQL | 结合参数分布和执行计划定位问题 |

## 2. 索引实验标准答案

## IDX-01：订单列表深分页

### 一句话结论

`LIMIT 10000,20` 需要先找到并跳过前 10000 行，再返回 20 行，深分页会放大索引扫描和回表成本。

### 现象解释

即使命中 `idx_user_id_created_at`，MySQL 也要按 `user_id` 和 `created_at` 的索引顺序扫描到 offset 位置。若 `SELECT *`，二级索引不能覆盖所有字段，还要根据主键回表读取完整订单行。

### 关键指标

看 `key` 是否为 `idx_user_id_created_at`，`rows` 是否接近 offset，`Extra` 是否没有覆盖索引，`EXPLAIN ANALYZE` 的实际行数和耗时是否随 offset 增长。

### 优化或处理方案

改游标分页；限制最大翻页深度；列表页减少字段；必要时使用 `(user_id, created_at, id)` 或 `(user_id, status, created_at, id)`；使用延迟回表先取 ID 再查详情。

### 面试表达

“深分页慢的本质不是返回 20 行慢，而是为了找到这 20 行要扫描并丢弃前面大量记录。二级索引如果不能覆盖返回字段，还会产生大量回表。我会优先把 offset 分页改成基于 created_at 和 id 的游标分页，再配合合适的联合索引和字段裁剪。”

### 常见误区

- 以为 `LIMIT 10000,20` 只读取 20 行。
- 只加单列索引，不考虑排序和回表。
- 直接上分库分表，忽略分页方式优化。

## IDX-02：覆盖索引与回表对比

### 一句话结论

只查询索引中已有的列时可以避免回表，`Extra` 可能出现 `Using index`。

### 现象解释

InnoDB 二级索引叶子节点保存索引列和主键值。`idx_user_id_created_at` 中包含 `user_id`、`created_at` 和主键 `id`，所以查询 `id,user_id,created_at` 可由索引直接返回；`SELECT *` 需要回表读取其他字段。

### 关键指标

对比两条 SQL 的 `Extra`：覆盖索引通常显示 `Using index`；`SELECT *` 通常没有该标记，且实际耗时可能更高。

### 优化或处理方案

列表页只返回必要字段；为高频查询设计覆盖索引；不要为了覆盖所有字段创建过宽索引。

### 面试表达

“覆盖索引的价值是减少回表。对列表页来说，先明确展示字段，再让联合索引覆盖过滤、排序和返回列，通常比盲目 SELECT * 稳定得多。”

### 常见误区

- 认为使用索引就一定没有回表。
- 为了覆盖索引把所有字段都塞进联合索引。
- 忽略写入成本和索引维护成本。

## IDX-03：联合索引顺序

### 一句话结论

订单列表索引要围绕等值过滤、排序、稳定分页设计，`(user_id, status, created_at, id)` 比 `(user_id, created_at)` 更贴近带状态的订单列表。

### 现象解释

`user_id` 是高频等值过滤；`status` 虽低选择性，但放在用户之后能继续缩小该用户订单范围；`created_at` 用于排序；`id` 用于同一时间下稳定排序和游标分页。

### 关键指标

新增索引前后对比 `key`、`rows`、`Extra`。理想情况是扫描行数下降，排序成本下降，返回字段更可能被覆盖。

### 优化或处理方案

根据真实 WHERE 和 ORDER BY 设计联合索引。等值字段靠前，排序字段随后；范围字段后面的列要谨慎判断是否还能用于排序。

### 面试表达

“我不会孤立地说 status 低选择性就不能建索引，而会看它在联合索引中的位置。对于用户订单列表，user_id 先定位用户，status 再过滤该用户订单，created_at 支撑排序，id 保证分页稳定。”

### 常见误区

- 机械套用“选择性最高字段一定放最前”。
- 不结合 ORDER BY 设计索引。
- 忽略同一 created_at 下排序不稳定问题。

## IDX-04：范围条件影响排序

### 一句话结论

联合索引中遇到范围条件后，后续列通常难以继续用于全局有序输出，可能出现 `Using filesort`。

### 现象解释

索引 `(user_id, created_at, total_amount)` 在 `user_id` 等值后，`created_at >= ...` 是范围扫描。范围内 `total_amount` 不是全局连续有序输出，因此 `ORDER BY total_amount` 往往还要额外排序。

### 关键指标

看 `Extra` 是否出现 `Using filesort`，以及 `rows` 是否较大。`key` 命中不代表排序也完全由索引完成。

### 优化或处理方案

如果业务核心排序是金额，可考虑 `(user_id, total_amount)` 或增加筛选条件；如果核心路径是时间列表，产品交互上优先支持按时间排序。

### 面试表达

“联合索引不是只看 WHERE 是否命中，还要看 ORDER BY 是否能利用索引顺序。范围条件常会截断后续列的有序性，所以看到 key 命中了仍然可能 filesort。”

### 常见误区

- 认为索引中包含排序字段就一定不 filesort。
- 不区分过滤利用索引和排序利用索引。
- 为少量后台查询设计影响写入的大索引。

## IDX-05：低选择性字段索引

### 一句话结论

低选择性字段单独建索引价值有限，但放入符合业务路径的联合索引中仍然可能有价值。

### 现象解释

`status = 3` 可能命中大量订单，单独过滤效果弱。但如果查询总是带 `tenant_id` 或 `user_id`，则 `tenant_id/status/created_at` 或 `user_id/status/created_at` 可以先定位业务范围，再过滤状态并支持排序。

### 关键指标

看不同参数下 `rows` 差异；热点租户可能扫描更多。关注优化器是否选择 `idx_status_created_at` 或其他更合适索引。

### 优化或处理方案

低选择性字段不要孤立判断；结合租户、用户、时间、排序建立联合索引；必要时做统计信息刷新。

### 面试表达

“status 这种字段单独看选择性很低，但在订单列表里它不是单独使用，而是和 tenant_id/user_id、created_at 组合使用。索引设计要服务具体查询路径，而不是只看单列基数。”

### 常见误区

- 一概认为低选择性字段不能进入索引。
- 忽略热点租户导致的数据倾斜。
- 不看真实参数分布。

## IDX-06：隐式类型转换

### 一句话结论

字符串列应使用字符串参数比较，否则可能发生隐式类型转换，导致索引使用异常或语义风险。

### 现象解释

`phone` 是 `VARCHAR`。当用数字常量比较时，MySQL 可能进行类型转换。若转换发生在字段侧，可能破坏索引的有序性利用；即使某些版本能优化，也不应依赖不稳定行为。

### 关键指标

对比 `phone = '13800001001'` 和 `phone = 13800001001` 的 `type`、`key`、`rows`，并查看 `SHOW WARNINGS`。

### 优化或处理方案

接口层、ORM、DAO 层统一手机号为字符串；SQL 审核中加入字段类型和参数类型一致性检查。

### 面试表达

“手机号字段我会用字符串存储和比较，因为手机号不是数学数值。隐式类型转换不仅可能影响索引，也可能带来前导零、国际区号、精度和语义问题。”

### 常见误区

- 把手机号当数字处理。
- 以为 EXPLAIN 某次命中索引就没有风险。
- 忽略 ORM 参数绑定类型。

## IDX-07：Using index 与 Using index condition

### 一句话结论

`Using index` 通常表示覆盖索引；`Using index condition` 表示索引下推，仍可能需要回表。

### 现象解释

只查 `category, price` 时，`idx_category_price` 可覆盖返回列，可能显示 `Using index`。`SELECT *` 时需要完整商品行，存储引擎可先在索引层过滤 `category/price`，再回表，因此可能显示 `Using index condition`。

### 关键指标

看 `Extra`，并结合返回字段判断是否回表。不要把 “Using index condition” 误认为覆盖索引。

### 优化或处理方案

减少返回列；为高频查询建立覆盖索引；避免把 ICP 当作万能优化。

### 面试表达

“Using index 是覆盖索引，不需要回表；Using index condition 是索引下推，它能减少回表前的无效记录，但最终返回非索引列时仍要回表。”

### 常见误区

- 把 `Using index condition` 当成覆盖索引。
- 认为 ICP 能消除所有回表。
- 忽略 SELECT 字段对执行计划的影响。

## IDX-08：JOIN 查询索引

### 一句话结论

JOIN 优化要看驱动表过滤行数、被驱动表关联索引和排序聚合成本。

### 现象解释

如果从 `order_items` 以 `product_id = 2001` 过滤，热点商品会返回大量明细，再通过订单主键回表查订单。若还要按 `o.created_at` 排序，可能产生额外排序。

### 关键指标

看两张表各自的 `type`、`key`、`rows` 和 `Extra`；关注驱动表是否过大，是否出现 `Using filesort`。

### 优化或处理方案

根据报表路径补索引；减少实时 JOIN 聚合；热点商品销量改预聚合；必要时将报表迁移到 OLAP。

### 面试表达

“JOIN 慢不是只看有没有 join 字段索引，还要看从哪张表开始、第一步过滤多少行、后续关联是否走主键、排序和聚合是否产生临时表。”

### 常见误区

- 只给 join 字段加索引，不看过滤和排序。
- 忽略热点商品导致驱动表过大。
- 实时报表全部压在线上库。

## 3. 锁实验标准答案

## LOCK-01：库存扣减行锁

### 一句话结论

主键等值更新会对命中的商品行加排他锁，其他事务更新同一行必须等待。

### 现象解释

`UPDATE products WHERE id = 2001` 通过主键定位单行，InnoDB 对该记录加 X 锁。会话 A 未提交时，会话 B 更新同一行会进入锁等待。

### 关键指标

`SHOW PROCESSLIST` 中会看到等待状态；`data_lock_waits` 可看到等待关系；`SHOW ENGINE INNODB STATUS` 可看事务持锁信息。

### 优化或处理方案

事务内只做必要 SQL，避免长事务；扣库存使用 `stock > 0` 原子条件；热点库存可用队列、预扣、分段库存削峰。

### 面试表达

“扣库存我会用一条带条件的 UPDATE 保证原子性，但热点商品会形成热点行锁，所以要控制事务时长，必要时引入队列或库存分片。”

### 常见误区

- 先 SELECT 再 UPDATE，导致并发超卖风险。
- 在事务里调用外部接口，长时间持锁。
- 看到 SQL 简单就忽略锁等待。

## LOCK-02：订单状态更新锁等待

### 一句话结论

并发更新同一订单会串行化，等待结束后会重新判断 WHERE 条件。

### 现象解释

会话 A 更新 `orders.id = 3001` 后持有该行 X 锁。会话 B 等待。A 提交后，如果订单状态已从 0 变为 1，B 的 `WHERE status = 0` 不再满足，影响 0 行。

### 关键指标

看锁等待链路、`ROW_COUNT()` 或 affected rows、最终订单状态。

### 优化或处理方案

状态机更新必须带旧状态条件；根据 affected rows 判断幂等成功、重复请求或状态冲突。

### 面试表达

“订单状态更新不是简单 UPDATE id，而是 UPDATE id AND old_status。这样并发请求即使串行执行，也能靠 affected rows 判断谁真正完成了状态流转。”

### 常见误区

- 不带旧状态条件，后来的请求覆盖前面的状态。
- 不检查 affected rows。
- 把锁等待误判为数据库性能差。

## LOCK-03：范围更新与间隙锁 / 临键锁

### 一句话结论

REPEATABLE READ 下范围更新可能加 next-key lock，锁住记录和间隙，阻塞范围内插入。

### 现象解释

范围条件通过索引扫描时，InnoDB 为避免当前读下的幻读，可能锁定索引记录以及记录之间的 gap。插入落在该范围内时，需要等待范围锁释放。

### 关键指标

看 `data_locks` 中的锁模式和索引名；看插入事务是否等待；用 `SHOW ENGINE INNODB STATUS` 查看锁详情。

### 优化或处理方案

避免大范围更新；拆小批次；降低隔离级别需谨慎评估；让范围条件尽量精确命中索引；离峰执行批处理。

### 面试表达

“范围更新的风险是锁范围可能远大于实际更新行数，尤其在 RR 下会涉及 gap/next-key lock。线上批处理要小批量、短事务、走精确索引。”

### 常见误区

- 认为 UPDATE 只锁实际修改的行。
- 忽略索引选择对锁范围的影响。
- 大事务批量更新线上热表。

## LOCK-04：唯一索引冲突等待

### 一句话结论

唯一键检查遇到未提交的相同唯一值时，需要等待对方事务结果。

### 现象解释

会话 A 插入手机号但未提交。会话 B 插入同手机号时不能立即判断最终是否冲突：如果 A 提交则冲突，如果 A 回滚则可成功，所以 B 会等待。

### 关键指标

看 `uk_phone` 相关锁等待；A 提交后 B 报重复键，A 回滚后 B 成功。

### 优化或处理方案

注册接口依赖唯一键兜底；捕获重复键异常；避免长事务包裹注册流程；必要时使用幂等 token。

### 面试表达

“唯一索引不仅用于查询，也承担并发唯一性约束。两个事务插入同一唯一值时，后来的事务要等先来的事务提交或回滚，才能决定成功还是报冲突。”

### 常见误区

- 以为重复唯一键一定立即报错。
- 只在应用层查重，不依赖数据库唯一约束。
- 忽略未提交事务对唯一性判断的影响。

## LOCK-05：死锁复现

### 一句话结论

两个事务按相反顺序持有并等待对方资源，会形成死锁，InnoDB 会回滚其中一个事务。

### 现象解释

A 先锁 2001 再等 2002；B 先锁 2002 再等 2001，形成循环等待。InnoDB 死锁检测发现环后，选择代价较小的事务回滚。

### 关键指标

`SHOW ENGINE INNODB STATUS` 的 LATEST DETECTED DEADLOCK；关注两条事务 SQL、持有锁、等待锁。

### 优化或处理方案

统一加锁顺序；缩短事务；降低一次事务锁定资源数量；捕获死锁错误并有限重试。

### 面试表达

“死锁不是 MySQL 出 bug，而是并发资源获取顺序形成环。我的处理思路是统一加锁顺序、缩短事务，并在业务层对可重试操作做幂等重试。”

### 常见误区

- 把死锁当成偶发不用管。
- 只增大 lock wait timeout。
- 没有业务幂等就盲目重试。

## LOCK-06：SELECT、FOR UPDATE、UPDATE 锁差异

### 一句话结论

普通 SELECT 是快照读，通常不阻塞；`SELECT ... FOR UPDATE` 和 UPDATE 是当前读，会读最新版本并加锁。

### 现象解释

会话 A 更新商品未提交时，普通 SELECT 可通过 MVCC 读取旧版本；`FOR UPDATE` 需要获取最新记录的排他锁或锁兼容性，因此等待 A 释放锁。

### 关键指标

普通 SELECT 是否立即返回；FOR UPDATE 是否等待；锁等待表中是否出现等待关系。

### 优化或处理方案

只展示数据用普通 SELECT；要基于读取结果做写入决策时使用当前读或直接条件 UPDATE。

### 面试表达

“快照读解决读写不阻塞，当前读解决并发修改的一致性。是否用 FOR UPDATE 取决于后续是否要基于这行数据做强一致写操作。”

### 常见误区

- 认为所有 SELECT 都不加锁。
- 在只读接口滥用 FOR UPDATE。
- 先普通 SELECT 再依赖结果更新，导致竞态。

## 4. 事务与 MVCC 标准答案

## TX-01：READ COMMITTED 两次读不同

### 一句话结论

READ COMMITTED 下每条语句通常生成新的 Read View，所以同一事务内两次快照读可能看到不同已提交版本。

### 现象解释

会话 A 第一次 SELECT 看到旧状态；会话 B 更新并提交；会话 A 第二次 SELECT 生成新的 Read View，于是看到新状态。

### 关键指标

两次 SELECT 的 `status` 是否变化；隔离级别是否为 READ COMMITTED。

### 优化或处理方案

如果业务需要事务内稳定视图，使用 REPEATABLE READ 或显式锁定；如果需要及时看到最新提交，READ COMMITTED 更合适。

### 面试表达

“RC 的特点是读已提交，每条语句看到的是语句开始前已经提交的数据，所以同一事务里两次普通 SELECT 可能不一致。”

### 常见误区

- 以为事务内所有隔离级别都能可重复读。
- 混淆不可重复读和脏读。

## TX-02：REPEATABLE READ 两次读相同

### 一句话结论

REPEATABLE READ 下事务内普通快照读通常复用同一个 Read View，因此两次读取相同。

### 现象解释

会话 A 第一次快照读创建 Read View。会话 B 后续提交的新版本对这个 Read View 不可见，所以 A 第二次普通 SELECT 仍看到旧版本。

### 关键指标

两次普通 SELECT 的 `status` 是否相同；提交后新事务是否能看到新状态。

### 优化或处理方案

需要稳定报表视图可用 RR；需要最新数据则使用当前读或缩短事务。

### 面试表达

“RR 的可重复读主要靠 MVCC Read View 实现，不是靠普通 SELECT 加锁。事务内一致性强，但也可能让你读不到其他事务刚提交的新值。”

### 常见误区

- 认为 RR 普通 SELECT 会锁住记录。
- 认为看不到新数据就是数据没提交。

## TX-03：快照读和当前读差异

### 一句话结论

同一事务中，普通 SELECT 读快照，`SELECT ... FOR UPDATE` 读最新可更新版本。

### 现象解释

普通 SELECT 受 Read View 限制；当前读用于后续修改，必须基于最新已提交版本，并加锁防止并发修改。

### 关键指标

普通 SELECT 和 FOR UPDATE 返回的 `status` 是否不同；是否产生锁等待。

### 优化或处理方案

不要混淆展示型查询和决策型查询；状态流转优先使用条件 UPDATE 或 FOR UPDATE。

### 面试表达

“MVCC 解决的是快照读一致性，但 UPDATE/DELETE/FOR UPDATE 这类当前读要操作最新记录，所以它们的可见性和普通 SELECT 不一样。”

### 常见误区

- 在事务里先普通 SELECT，以为后续 UPDATE 一定基于这个旧结果。
- 不理解当前读为什么会看到新版本。

## TX-04：UPDATE 当前读

### 一句话结论

UPDATE 会按最新可更新版本判断 WHERE 条件，而不是按事务早期快照判断。

### 现象解释

会话 A 早期快照看到 `status=0`，但会话 B 提交后最新版本为 `status=1`。A 执行 `UPDATE ... WHERE status=1` 可以命中新版本，说明 UPDATE 是当前读。

### 关键指标

`ROW_COUNT()` 是否为 1；最终订单状态是否更新为目标状态。

### 优化或处理方案

状态机更新用 `WHERE id=? AND status=?`；用 affected rows 做并发和幂等判断。

### 面试表达

“UPDATE 是当前读，所以并发状态流转要把旧状态写在 WHERE 里，让数据库帮我原子判断当前最新状态是否仍满足条件。”

### 常见误区

- 以为 UPDATE 会使用普通 SELECT 的旧快照。
- 不检查 affected rows。

## TX-05：幻读观察

### 一句话结论

RC 下普通范围查询可能看到新插入行；RR 下普通快照读通常保持一致，但当前读范围锁另当别论。

### 现象解释

RC 每次语句新建 Read View，第二次 COUNT 能看到 B 插入并提交的新行；RR 普通快照读复用 Read View，因此看不到新行。若使用 `FOR UPDATE`，会涉及 next-key lock 防止范围内插入。

### 关键指标

两次 COUNT 是否变化；隔离级别；是否使用当前读。

### 优化或处理方案

报表统计用一致性快照；需要防止并发插入影响业务决策时使用当前读或唯一约束。

### 面试表达

“幻读关注的是范围内新增或删除行。RR 下普通快照读通过 Read View 保持一致，而当前读为了防止幻影行可能使用间隙锁或临键锁。”

### 常见误区

- 简单说 RR 绝对没有幻读，不区分快照读和当前读。
- 把不可重复读和幻读混为一谈。

## TX-06：订单状态流转事务

### 一句话结论

订单状态更新和日志写入应在同一事务中完成，并用旧状态条件保证幂等。

### 现象解释

两个事务同时尝试从 `status=0` 改走，只有一个能成功。另一个等待后发现状态已变化，affected rows 为 0。日志与订单更新同事务提交，保证一致性。

### 关键指标

`ROW_COUNT()`、最终 `orders.status`、`order_logs` 是否写入、是否有锁等待。

### 优化或处理方案

条件 UPDATE；日志同事务；重复回调用幂等键或状态机判断；失败请求不盲目重试覆盖状态。

### 面试表达

“订单支付不是先查再改，而是用一条条件 UPDATE 抢状态流转权，成功后同事务写日志。这样并发支付、取消、重复回调都能通过 affected rows 做幂等处理。”

### 常见误区

- 订单和日志分两个事务提交。
- 不带旧状态导致状态覆盖。
- 重复回调直接报错，不做幂等。

## 5. 慢查询排查标准答案

## SLOW-01：订单列表深分页

### 一句话结论

深分页慢的核心是 offset 扫描和回表成本，而不是返回行数本身。

### 现象解释

热点用户订单多，`LIMIT 10000,20` 要扫描并丢弃大量索引记录。`SELECT *` 还会把大量候选记录回表读取完整行。

### 关键指标

`rows` 接近 offset；`EXPLAIN ANALYZE` 实际扫描多；列表页耗时随页码增长。

### 优化或处理方案

先限制翻页和字段；再改游标分页；补 `(user_id, created_at, id)`；延迟回表；历史归档。

### 面试表达

“我会先用 EXPLAIN ANALYZE 证明它扫描了 offset + size 级别的数据，然后把接口从 offset 分页改成基于游标的 seek pagination，并控制列表字段。”

### 常见误区

- 只调大数据库参数。
- 直接分表。
- 不改分页方式。

## SLOW-02：热点租户扫描过大

### 一句话结论

同一 SQL 在热点租户下慢，是数据倾斜导致扫描范围和过滤成本放大。

### 现象解释

`tenant_id=1` 占订单大多数，现有 `(tenant_id, created_at)` 不能充分利用 `status` 过滤，可能扫描大量该租户订单再过滤。

### 关键指标

租户分布统计；`rows`；`key` 是否缺少 status；热点租户和普通租户耗时对比。

### 优化或处理方案

补 `(tenant_id, status, created_at, id)`；限制时间范围；热点租户缓存；长期考虑分区、归档、分表。

### 面试表达

“数据倾斜会让平均耗时掩盖热点参数问题。我会先按 tenant_id 统计分布，再针对高频查询路径设计 tenant_id/status/created_at 的联合索引。”

### 常见误区

- 只看平均耗时。
- 不分析参数分布。
- 低选择性 status 一概不入索引。

## SLOW-03：日志时间范围查询

### 一句话结论

日志表慢通常来自数据持续增长、时间范围过大、深分页和 SELECT * 回表。

### 现象解释

`idx_created_at` 能帮助时间扫描，但最近 30 天日志仍可能很多。深分页继续放大 offset，`SELECT *` 放大回表成本。

### 关键指标

`rows`、实际扫描行数、offset、返回字段、是否覆盖索引。

### 优化或处理方案

限制时间范围；游标分页；覆盖索引；异步导出；日志归档、分区、冷热分离。

### 面试表达

“日志查询不能只靠 created_at 单列索引无限撑下去。我要控制查询窗口和分页方式，线上只查热数据，历史导出走归档或离线链路。”

### 常见误区

- 让线上接口支持任意时间范围导出。
- 认为有 created_at 索引就不会慢。
- 忽略日志表增长趋势。

## SLOW-04：JOIN 聚合报表

### 一句话结论

实时 JOIN + GROUP BY + ORDER BY 在大表上很容易产生大量扫描、临时表和 filesort。

### 现象解释

订单过滤、明细关联、商品分组、销量排序分属不同阶段。即使关联字段有索引，也可能因为聚合排序产生临时表和额外排序。

### 关键指标

`Using temporary`、`Using filesort`、驱动表 rows、JOIN 后行数、聚合耗时。

### 优化或处理方案

缩小范围；缓存报表；补过滤和关联索引；建立日维度汇总表；迁移 OLAP。

### 面试表达

“报表慢我不会只说加索引，而是先拆解过滤、关联、聚合、排序四个阶段。高频销量榜应该预聚合，而不是每次扫明细实时算。”

### 常见误区

- 只给 GROUP BY 字段建索引。
- 强行要求 OLTP 库做复杂分析。
- 忽略报表实时性要求可以降级。

## SLOW-05：隐式类型转换

### 一句话结论

字段类型和参数类型不一致会导致隐式转换，可能让索引失效或行为不稳定。

### 现象解释

手机号是字符串，数字参数可能触发类型转换。最佳实践是应用层严格用字符串绑定参数。

### 关键指标

对比两条 EXPLAIN；查看 `SHOW WARNINGS`；检查慢日志中的 SQL 文本和参数类型。

### 优化或处理方案

修复接口参数；统一 schema 与代码类型；SQL 审核；核心 SQL 巡检。

### 面试表达

“这类问题不是靠给 phone 再加索引解决，而是要修正参数类型。索引能否稳定命中，前提是比较表达式本身对索引友好。”

### 常见误区

- 把手机号设为 BIGINT。
- 只看一次 EXPLAIN 命中就放心。
- 不关注参数绑定类型。

## SLOW-06：锁等待导致 SQL 慢

### 一句话结论

执行计划好不代表 SQL 一定快，锁等待会让主键更新也执行很久。

### 现象解释

`UPDATE products WHERE id=2001` 扫描行数很小，但如果目标行被其他长事务持有 X 锁，就只能等待。

### 关键指标

`SHOW PROCESSLIST` 等待状态；`data_lock_waits` 阻塞链路；长事务开始时间；InnoDB status。

### 优化或处理方案

止血：确认后提交、回滚或 kill 阻塞事务。长期：缩短事务、固定加锁顺序、热点行削峰、长事务监控。

### 面试表达

“慢 SQL 排查不能只看 EXPLAIN。主键更新如果慢，我会立刻查 processlist 和 data_lock_waits，确认是不是锁等待，再追阻塞源事务。”

### 常见误区

- 看到主键命中就排除数据库问题。
- 只优化索引，不查锁。
- 用增大超时时间掩盖长事务。

## 6. 面试可背诵版本

## 6.1 深分页

深分页慢的核心是 offset 扫描。`LIMIT 10000,20` 不是只读 20 行，而是要先定位并跳过前 10000 行。优化上优先改游标分页，其次做字段裁剪、覆盖索引和延迟回表。

## 6.2 联合索引设计

联合索引要围绕具体查询设计。一般先放等值过滤字段，再放排序或范围字段，最后可以放主键保证稳定分页。不能只按单列选择性机械排序，还要看 WHERE、ORDER BY、LIMIT 和返回字段。

## 6.3 覆盖索引

覆盖索引是指查询所需字段都能从索引中得到，不需要回表。`Using index` 通常表示覆盖索引。它适合高频列表页，但不能为了覆盖而无限加宽索引。

## 6.4 Using index condition

`Using index condition` 是索引下推，表示存储引擎先用索引条件过滤一部分记录，再回表。它能减少无效回表，但不是覆盖索引。

## 6.5 行锁和锁等待

InnoDB 行锁通常加在索引记录上。主键等值更新会锁定命中的记录，其他事务更新同一行会等待。锁等待慢要用 `SHOW PROCESSLIST`、`data_locks`、`data_lock_waits` 和 `SHOW ENGINE INNODB STATUS` 排查。

## 6.6 死锁

死锁是多个事务互相等待对方持有的锁，形成循环等待。处理方式是统一加锁顺序、缩短事务、减少一次锁定资源数量，并在业务可幂等的前提下有限重试。

## 6.7 RC 与 RR

READ COMMITTED 每条语句通常生成新的 Read View，所以同一事务内两次读可能不同。REPEATABLE READ 普通快照读通常复用 Read View，所以两次读一致。

## 6.8 快照读与当前读

普通 SELECT 是快照读，读的是 Read View 可见版本。`SELECT ... FOR UPDATE`、`UPDATE`、`DELETE` 是当前读，读取最新可更新版本，并可能加锁。

## 6.9 慢查询排查路径

慢查询排查要先定位影响面、慢 SQL、参数分布和执行计划，再做 SQL 改写、索引优化、查询路径优化，最后才考虑缓存、异步、读写分离、分区和分库分表。

## 6.10 锁等待型慢 SQL

执行计划很好但 SQL 很慢，要怀疑锁等待。典型表现是主键更新 rows 很小，但线程长时间 waiting。此时应该查阻塞链路，而不是盲目加索引。
