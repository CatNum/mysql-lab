# MySQL 模拟实战内容生成 Spec

> 用途：这份 spec 用来指导网页 ChatGPT 分文件生成 `README.md` 中列出的 MySQL 模拟实战材料。每个文件的说明都尽量独立，方便单独复制某一章给网页 ChatGPT 生成对应脚本或题目。

## 第一章：各个文件的关联

### 1.1 总体目标

这一组文件要设计成一个完整的 MySQL 面试实战实验室，而不是零散题目集合。所有内容统一围绕一个电商订单业务域展开：用户、商品、订单、订单明细、订单日志。

核心目标是让学习者能从同一批表和同一批数据出发，依次练习：

- 建表设计：理解字段、索引、约束、业务关系。
- 造数设计：构造真实的数据规模、数据分布、热点数据和异常数据。
- 索引实验：观察联合索引、覆盖索引、回表、深分页、排序、隐式类型转换。
- 锁实验：观察行锁、间隙锁、临键锁、死锁、库存扣减冲突。
- 事务与 MVCC 实验：观察隔离级别、快照读、当前读、幻读、版本可见性。
- 慢查询排查实验：模拟线上订单列表、报表、日志查询变慢，并用执行计划定位。
- 标准答案与观测指标：沉淀每个实验的预期现象、关键命令、面试表达。

### 1.4 统一业务数据模型

所有文件使用同一个数据库：

- `interview_lab`：面试实验库，用来承载全部 MySQL 模拟实战表。

所有文件统一使用五张表：

- `users`（用户表）：保存用户基础信息，用于手机号、邮箱、城市、年龄、状态、软删除等查询实验。
- `products`（商品表）：保存商品基础信息和库存，用于价格范围、分类查询、库存扣减和锁冲突实验。
- `orders`（订单表）：保存订单主数据，是索引、深分页、事务、慢查询的核心表。
- `order_items`（订单明细表）：保存订单和商品的多对多明细，用于 JOIN、聚合、报表慢查询实验。
- `order_logs`（订单日志表）：保存订单状态变更日志，用于日志深分页、时间范围扫描、归档类慢查询实验。

关键字段含义必须在生成内容里解释清楚：

- `id`（主键 ID）：唯一标识一行数据。
- `tenant_id`（租户 ID）：模拟多租户或大客户数据隔离，也用于制造数据倾斜。
- `user_id`（用户 ID）：关联 `users.id`，用于订单归属和用户订单列表查询。
- `product_id`（商品 ID）：关联 `products.id`，用于订单明细和库存实验。
- `order_id`（订单 ID）：关联 `orders.id`，用于订单明细、订单日志查询。
- `order_no`（订单号）：业务唯一标识，用于唯一索引和点查。
- `status`（状态）：表示用户、商品或订单状态，用于低选择性字段索引实验。
- `deleted`（软删除标记）：表示数据是否逻辑删除，用于复合索引和查询条件设计。
- `created_at`（创建时间）：用于时间范围、排序、分页和深分页实验。
- `updated_at`（更新时间）：用于更新观察和事务实验。
- `stock`（库存数量）：用于扣减库存、行锁、死锁实验。
- `total_amount`（订单总金额）：用于金额范围查询、聚合和报表实验。
- `pay_time`（支付时间）：用于支付订单筛选和时间范围实验。

## 第二章：`00.建表脚本.md` 生成说明

### 2.1 文件定位

`00.建表脚本.md` 是所有实验的基础。它负责创建数据库、定义表结构、字段注释、基础索引和必要的实验索引，不负责造数，不负责写实验题。造数脚本不创建数据库和表结构，只连接已经建好的 `interview_lab` 写入数据。

生成时要保证后续实验能使用同一套表完成，不要每个实验重新设计一套表。

### 2.2 必须包含的表

必须包含以下五张表：

- `users`（用户表）：用于用户维度查询、手机号隐式类型转换、邮箱唯一键、城市年龄组合索引。
- `products`（商品表）：用于商品分类价格查询、库存扣减、库存锁冲突。
- `orders`（订单表）：用于订单列表、深分页、订单状态查询、事务可见性、慢查询排查。
- `order_items`（订单明细表）：用于订单和商品关联、JOIN 查询、聚合报表。
- `order_logs`（订单日志表）：用于状态变更日志、时间范围查询、日志深分页。

### 2.3 索引设计原则

建表脚本里要保留一部分“合理但不完美”的索引，用来制造面试实验空间。

建议保留的索引：

- `users.uk_phone`（手机号唯一索引）：用于点查和隐式类型转换实验。
- `users.uk_email`（邮箱唯一索引）：用于唯一键和软删除组合讨论。
- `users.idx_city_age`（城市 + 年龄联合索引）：用于等值条件加范围条件实验。
- `orders.uk_order_no`（订单号唯一索引）：用于订单点查。
- `orders.idx_tenant_created_at`（租户 + 创建时间联合索引）：用于多租户订单列表。
- `orders.idx_user_id_created_at`（用户 ID + 创建时间联合索引）：用于用户订单列表和深分页。
- `orders.idx_status_created_at`（状态 + 创建时间联合索引）：用于低选择性字段加排序实验。
- `orders.idx_created_at`（创建时间索引）：用于时间范围扫描和优化器选择实验。
- `order_items.idx_order_id`（订单 ID 索引）：用于订单查明细。
- `order_items.idx_product_id`（商品 ID 索引）：用于商品销量聚合。
- `order_logs.idx_order_id_created_at`（订单 ID + 创建时间联合索引）：用于订单日志查询。
- `order_logs.idx_created_at`（创建时间索引）：用于日志时间范围和深分页。

建议故意缺失的索引：

- 缺少 `orders(user_id, status, created_at, id)`，用于让索引实验题引导学习者设计更合适的订单列表索引。
- 缺少覆盖订单列表返回字段的索引，用于观察 `SELECT *` 回表。
- 缺少 `order_items(product_id, created_at)`，用于制造商品销量报表慢查询。

## 第三章：`01.造数脚本.md` 生成说明

### 3.1 文件定位

`01.造数脚本.md` 负责说明实验数据如何构造，真实可执行脚本放在 `../01.scripts/seed_interview_lab.py`。它要让后续索引、锁、事务、慢查询实验都能跑出明显现象。

造数脚本不要只插入几十行样例数据。至少要设计成可以生成万级到百万级数据，并允许调整规模。

### 3.2 数据规模建议

建议默认规模：

- `users`（用户表）：50,000 行。
- `products`（商品表）：5,000 行。
- `orders`（订单表）：500,000 行。
- `order_items`（订单明细表）：1,000,000 到 1,500,000 行。
- `order_logs`（订单日志表）：1,500,000 到 2,500,000 行。

造数方式直接定死为 Python 脚本，不使用 MySQL 存储过程。原因是这套实验需要构造百万级数据、热点用户、热点租户、热点商品、状态流转和订单日志关联，Python 更适合表达复杂数据分布，也更方便分批提交、显示进度和后续调整。

Python 脚本建议包含：

- 数据库连接配置：例如 host（数据库地址）、port（数据库端口）、user（用户名）、password（密码）、database（数据库名）。
- 批量插入函数：例如 `batch_insert`（批量插入函数，用于按批次写入多行数据，避免单次 SQL 太大）。
- 进度输出函数：例如 `print_progress`（进度输出函数，用于显示当前生成到哪张表、多少行）。
- 固定数据初始化函数：例如 `seed_fixed_records`（固定实验数据初始化函数，用于保证热点用户、热点商品、固定订单存在）。
- 普通数据生成函数：例如 `generate_users`（生成用户数据函数）、`generate_products`（生成商品数据函数）、`generate_orders`（生成订单数据函数）、`generate_order_items`（生成订单明细数据函数）、`generate_order_logs`（生成订单日志数据函数）。
- 数据校验函数：例如 `verify_counts`（数据量校验函数，用于检查每张表行数和热点数据是否达标）。

### 3.3 数据分布要求

必须构造以下数据特征：

- 热点租户：`tenant_id = 1` 占 60% 到 70% 的订单，用于模拟大客户数据倾斜。
- 普通租户：其他 `tenant_id` 分散在 2 到 20 之间。
- 热点用户：少数 `user_id` 拥有大量订单，用于用户订单列表深分页。
- 普通用户：大部分用户只有少量订单，用于对比不同参数下执行计划差异。
- 热点商品：少数 `product_id` 出现在大量订单明细中，用于销量聚合慢查询。
- 状态分布倾斜：订单 `status` 不能平均分布，已完成订单占比最高，待支付和取消占比较低。
- 时间分布：`created_at` 横跨最近 180 天，其中最近 7 天数据更多，用于时间范围查询。
- 可锁定数据：固定保留几个 `products.id` 作为库存扣减实验对象，库存值不能为 0。
- 可事务观察数据：固定保留几个 `orders.id` 和 `users.id`，方便不同会话反复查询和更新。

### 3.4 必须输出的固定实验数据

造数脚本必须保证以下数据存在，便于后续题目写死 SQL：

- `users.id = 1001`：热点用户，至少 20,000 笔订单。
- `users.id = 1002`：普通用户，约 20 到 100 笔订单。
- `products.id = 2001`：热点商品，库存大于 1,000，用于库存扣减和锁冲突。
- `products.id = 2002`：普通商品，库存大于 100，用于对照实验。
- `orders.id = 3001`：固定订单，状态为待支付，用于事务更新实验。
- `orders.id = 3002`：固定订单，状态为已支付，用于当前读和快照读实验。
- `tenant_id = 1`：热点租户，至少 300,000 笔订单。

## 第四章：`02.索引实验题.md` 生成说明

### 4.1 文件定位

`02.索引实验题.md` 负责设计可执行的索引实验题。每道题必须有题目背景、待执行 SQL、观察命令、思考问题，但标准答案可以只写简版，完整答案放到 `06.标准答案与观测指标.md`。

### 4.2 必须覆盖的实验

建议至少包含 8 道题：

- `IDX-01`：订单列表深分页，观察 `LIMIT 10000,20` 扫描和回表。
- `IDX-02`：覆盖索引对比，比较 `SELECT *` 和只查索引列。
- `IDX-03`：联合索引顺序，比较 `(user_id, created_at)` 和 `(user_id, status, created_at, id)`。
- `IDX-04`：范围条件影响排序，观察范围字段后面的排序字段是否还能利用索引。
- `IDX-05`：低选择性字段索引，观察 `status` 单独或组合索引的效果。
- `IDX-06`：隐式类型转换，比较 `phone = '138...'` 和 `phone = 138...`。
- `IDX-07`：`Using index` 与 `Using index condition` 区别。
- `IDX-08`：JOIN 查询索引，观察 `orders` 与 `order_items` 关联时的驱动表和索引选择。

### 4.3 每道题的固定结构

每道题都使用以下结构：

```text
## IDX-xx：题目名称

### 场景
说明这是哪个真实业务场景。

### 目标
说明要观察的 MySQL 机制。

### 准备
说明依赖哪些表、哪些固定数据、是否需要新增或删除索引。

### 执行 SQL
给出可直接执行的 SQL。

### 观察命令
给出 EXPLAIN / EXPLAIN ANALYZE / SHOW WARNINGS 等命令。

### 思考问题
列出 3 到 5 个面试追问。

### 记录区
预留学习者填写 type、key、rows、filtered、Extra、耗时、结论。
```

字段解释要求：

- `type`（访问类型）：说明 MySQL 访问表的方式。
- `key`（实际使用索引）：说明优化器选择了哪个索引。
- `rows`（预估扫描行数）：说明优化器预计扫描多少行。
- `filtered`（过滤比例）：说明扫描后预计保留多少比例。
- `Extra`（额外信息）：说明是否出现覆盖索引、索引下推、文件排序、临时表。
