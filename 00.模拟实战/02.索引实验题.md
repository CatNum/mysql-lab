# 02.索引实验题

> 目标：基于 `interview_lab` 的五张表练习 MySQL 8.0 索引定位能力。每道题都包含场景、目标、准备、执行 SQL、观察命令、思考问题和记录区。完整标准答案放在 `06.标准答案与观测指标.md`。

## 0. EXPLAIN 字段速查

| 字段 | 含义 | 观察重点 |
|---|---|---|
| `type` | 访问类型，表示 MySQL 如何访问表，例如 `const`、`ref`、`range`、`index`、`ALL` | 越接近 `const/ref/range` 通常越好，`ALL` 表示全表扫描 |
| `key` | 实际使用的索引 | 是否命中预期索引 |
| `rows` | 预估扫描行数 | 是否明显大于返回行数 |
| `filtered` | 过滤比例 | 扫描后预计保留多少比例 |
| `Extra` | 额外信息 | 是否出现 `Using index`、`Using index condition`、`Using filesort`、`Using temporary` |

## IDX-01：订单列表深分页

### 场景

用户打开“我的订单”第 500 页，接口使用传统 `LIMIT offset,size` 分页。热点用户 `users.id = 1001` 至少有 20,000 笔订单。

### 目标

观察深分页下扫描行数变大、回表成本变高，以及为什么 `LIMIT 10000,20` 不是只读取 20 行。

### 准备

依赖 `orders.idx_user_id_created_at(user_id, created_at)` 和热点用户数据。

### 执行 SQL

```sql
USE interview_lab;

SELECT *
FROM orders
WHERE user_id = 1001
ORDER BY created_at DESC
LIMIT 10000, 20;
```

### 观察命令

```sql
EXPLAIN SELECT *
FROM orders
WHERE user_id = 1001
ORDER BY created_at DESC
LIMIT 10000, 20;

EXPLAIN ANALYZE SELECT *
FROM orders
WHERE user_id = 1001
ORDER BY created_at DESC
LIMIT 10000, 20;
```

### 思考问题

1. `rows` 为什么可能接近 offset + size？
2. `SELECT *` 为什么更容易产生回表？
3. 深分页除了加索引，还能怎么优化？
4. 游标分页为什么通常比 offset 分页更稳定？

### 简短提示

先观察 `key` 是否使用 `idx_user_id_created_at`，再观察 `rows` 和实际耗时。

## IDX-03：联合索引顺序对订单列表的影响

### 场景

订单列表常见查询条件是用户 ID、订单状态、创建时间倒序。现有索引只有 `(user_id, created_at)`，没有把 `status` 放进去。

### 目标

观察联合索引顺序对过滤、排序和扫描行数的影响。

### 准备

先观察现有索引，再临时新增更贴近查询的索引。

```sql
SHOW INDEX FROM orders;
```

### 执行 SQL

```sql
EXPLAIN SELECT id, user_id, status, created_at
FROM orders
WHERE user_id = 1001
  AND status = 3
ORDER BY created_at DESC, id DESC
LIMIT 20;

ALTER TABLE orders
ADD INDEX idx_user_status_created_id (user_id, status, created_at, id);

EXPLAIN SELECT id, user_id, status, created_at
FROM orders
WHERE user_id = 1001
  AND status = 3
ORDER BY created_at DESC, id DESC
LIMIT 20;
```

实验后如需恢复：

```sql
ALTER TABLE orders DROP INDEX idx_user_status_created_id;
```

### 观察命令

```sql
EXPLAIN ANALYZE SELECT id, user_id, status, created_at
FROM orders
WHERE user_id = 1001
  AND status = 3
ORDER BY created_at DESC, id DESC
LIMIT 20;
```

### 思考问题

1. 为什么 `user_id` 适合作为联合索引第一列？
2. `status` 是低选择性字段，为什么放在 `user_id` 后面仍可能有价值？
3. `created_at` 同时用于排序时，索引顺序怎么设计？
4. `id` 放在最后有什么作用？

### 简短提示

等值条件优先，排序字段随后，主键 ID 可用于稳定排序和游标分页。

## IDX-04：范围条件影响排序

### 场景

运营后台按用户、时间范围查询订单，并按金额排序。

### 目标

观察联合索引中范围字段之后的排序字段通常难以继续用于有序输出。

### 准备

临时创建索引：

```sql
ALTER TABLE orders
ADD INDEX idx_user_created_amount (user_id, created_at, total_amount);
```

### 执行 SQL

```sql
EXPLAIN SELECT id, user_id, total_amount, created_at
FROM orders
WHERE user_id = 1001
  AND created_at >= DATE_SUB(NOW(), INTERVAL 180 DAY)
ORDER BY total_amount DESC
LIMIT 20;
```

实验后如需恢复：

```sql
ALTER TABLE orders DROP INDEX idx_user_created_amount;
```

### 观察命令

```sql
EXPLAIN ANALYZE SELECT id, user_id, total_amount, created_at
FROM orders
WHERE user_id = 1001
  AND created_at >= DATE_SUB(NOW(), INTERVAL 180 DAY)
ORDER BY total_amount DESC
LIMIT 20;
```

### 思考问题

1. 为什么 `created_at` 是范围条件后，`total_amount` 排序不一定能利用索引顺序？
2. `Extra` 中是否出现 `Using filesort`？
3. 如果业务必须按金额排序，索引如何重新设计？
4. 是否可以改成交互上只支持按时间排序？

### 简短提示

联合索引遵循最左前缀，范围条件常会截断后续列的有序利用能力。

## IDX-07：Using index 与 Using index condition

### 场景

同样是“使用索引”，`Extra` 里可能出现 `Using index`，也可能出现 `Using index condition`。

### 目标

区分覆盖索引和索引下推（Index Condition Pushdown, ICP）。

### 准备

依赖 `products.idx_category_price(category, price)`。

### 执行 SQL

```sql
EXPLAIN SELECT category, price
FROM products
WHERE category = '数码配件'
  AND price BETWEEN 100 AND 500
LIMIT 20;

EXPLAIN SELECT *
FROM products
WHERE category = '数码配件'
  AND price BETWEEN 100 AND 500
LIMIT 20;
```

### 观察命令

```sql
EXPLAIN ANALYZE SELECT category, price
FROM products
WHERE category = '数码配件'
  AND price BETWEEN 100 AND 500
LIMIT 20;

EXPLAIN ANALYZE SELECT *
FROM products
WHERE category = '数码配件'
  AND price BETWEEN 100 AND 500
LIMIT 20;
```

### 思考问题

1. `Using index` 表示什么？
2. `Using index condition` 表示什么？
3. 为什么 `SELECT *` 更容易回表？
4. ICP 能减少什么成本？不能减少什么成本？

### 简短提示

`Using index` 通常表示覆盖索引；`Using index condition` 通常表示先在索引层做部分条件过滤，再回表。

## IDX-08：JOIN 查询索引与驱动表

### 场景

报表接口查询热点商品对应的订单明细和订单信息。

### 目标

观察 JOIN 查询中驱动表选择、关联字段索引和过滤条件对执行计划的影响。

### 准备

依赖：

- `order_items.idx_product_id(product_id)`：商品 ID 索引。
- `orders.PRIMARY(id)`：订单主键。

### 执行 SQL

```sql
EXPLAIN SELECT o.id, o.order_no, o.user_id, oi.product_id, oi.quantity
FROM order_items oi
JOIN orders o ON o.id = oi.order_id
WHERE oi.product_id = 2001
  AND o.created_at >= DATE_SUB(NOW(), INTERVAL 30 DAY)
ORDER BY o.created_at DESC
LIMIT 50;
```

可对比：

```sql
EXPLAIN SELECT STRAIGHT_JOIN o.id, o.order_no, o.user_id, oi.product_id, oi.quantity
FROM orders o
JOIN order_items oi ON oi.order_id = o.id
WHERE oi.product_id = 2001
  AND o.created_at >= DATE_SUB(NOW(), INTERVAL 30 DAY)
ORDER BY o.created_at DESC
LIMIT 50;
```

### 观察命令

```sql
EXPLAIN ANALYZE SELECT o.id, o.order_no, o.user_id, oi.product_id, oi.quantity
FROM order_items oi
JOIN orders o ON o.id = oi.order_id
WHERE oi.product_id = 2001
  AND o.created_at >= DATE_SUB(NOW(), INTERVAL 30 DAY)
ORDER BY o.created_at DESC
LIMIT 50;
```

### 思考问题

1. 优化器选择哪张表作为驱动表？为什么？
2. `order_items.product_id` 和 `orders.id` 的索引分别起什么作用？
3. `ORDER BY o.created_at` 是否会引入 filesort？
4. 如果这是高频报表，是否应该补 `order_items(product_id, created_at)` 或做预聚合？

### 简短提示

JOIN 优化要同时看驱动表过滤行数、被驱动表关联索引、排序字段是否可利用索引。
